# 绩效评分指南 - 领导使用手册

## 📋 概述

本指南帮助技术管理者基于多人多分支对比报告，科学、公平地评估团队成员的工作表现。

## 🎯 核心评分维度

### 一、工程价值得分 (40%)

**指标来源**: 报告中的"平均得分"和"总得分"

#### 评分标准

| 平均得分 | 等级 | 基础分 | 说明 |
|---------|------|--------|------|
| 90+ | S | 40 | 卓越：顶尖工程师水准 |
| 80-89 | A | 35 | 优秀：高质量产出 |
| 70-79 | B | 30 | 良好：符合标准 |
| 60-69 | C | 25 | 合格：需改进 |
| <60 | D | 20 | 需提升：重点关注 |

#### 评估要点

```
✅ 看平均得分：反映每次提交的质量
✅ 看总得分：反映整体贡献量
✅ 综合评估：高质量 + 高产量 = 优秀
```

**示例**：
- **张三**: 平均得分 82.3 (A级) → 基础分 35
- **李四**: 平均得分 84.0 (A级) → 基础分 35
- **王五**: 平均得分 77.2 (B级) → 基础分 30

---

### 二、代码质量 (30%)

**指标来源**: 报告中的"代码质量对比"

#### 评分标准

| 质量得分 | 圈复杂度 | 认知复杂度 | 基础分 | 说明 |
|---------|---------|-----------|--------|------|
| 80+ | <15 | <20 | 30 | 优秀：架构清晰、易维护 |
| 70-79 | 15-20 | 20-30 | 25 | 良好：代码规范 |
| 60-69 | 20-25 | 30-40 | 20 | 合格：需优化 |
| <60 | >25 | >40 | 15 | 需改进：技术债务高 |

#### 评估要点

```
💎 质量得分：0-100，越高越好
🔍 圈复杂度：<15简单，>20复杂，>30需重构
🧠 认知复杂度：<20易读，>30难理解，>50需重写
```

**加分项**：
- 质量得分 85+ → +3 分
- 复杂度控制优秀（圈<12，认知<15）→ +2 分

**减分项**：
- 质量得分 <60 → -3 分
- 复杂度过高（圈>25 或 认知>40）→ -2 分

**示例**：
- **张三**: 质量 78, 圈 12, 认知 15 → 25 + 2 = 27 分
- **李四**: 质量 82, 圈 10, 认知 13 → 25 + 3 + 2 = 30 分
- **王五**: 质量 75, 圈 14, 认知 17 → 25 分

---

### 三、工作量 (20%)

**指标来源**: 报告中的"Commit数"和"代码量"

#### 评分标准

**3.1 提交数量** (10%)

根据团队平均值评估：

```javascript
个人占比 = 个人Commit数 / 团队平均Commit数

- 占比 ≥ 120% → 10 分
- 占比 100-119% → 8 分
- 占比 80-99% → 6 分
- 占比 60-79% → 4 分
- 占比 <60% → 2 分
```

**3.2 代码量** (10%)

根据团队平均值评估：

```javascript
个人占比 = 个人代码行数 / 团队平均代码行数

- 占比 ≥ 120% → 10 分
- 占比 100-119% → 8 分
- 占比 80-99% → 6 分
- 占比 60-79% → 4 分
- 占比 <60% → 2 分
```

#### 评估要点

```
⚖️ 不要只看绝对值，要看相对贡献
📊 考虑任务难度：核心模块 vs 简单功能
🔄 考虑角色定位：架构师 vs 业务开发
```

**示例**（假设团队平均 100 次提交，10000 行代码）：
- **张三**: 120 次, 15234 行 → 10 + 10 = 20 分
- **李四**: 98 次, 12456 行 → 8 + 10 = 18 分
- **王五**: 56 次, 6789 行 → 2 + 4 = 6 分

---

### 四、技术影响力 (10%)

**指标来源**: 报告中的"工程质量对比"

#### 评分标准

| 新增函数 | 新增类 | 新增接口 | React Hooks | 基础分 | 说明 |
|---------|--------|---------|------------|--------|------|
| >200 | >30 | >80 | >20 | 10 | 卓越：技术产出丰富 |
| 100-200 | 15-30 | 40-80 | 10-20 | 8 | 优秀：架构能力强 |
| 50-100 | 5-15 | 20-40 | 5-10 | 6 | 良好：稳定产出 |
| <50 | <5 | <20 | <5 | 4 | 需提升：产出较少 |

#### 评估要点

```
🏗️ 类和接口：体现架构设计能力
⚙️ 函数：体现功能实现能力
⚛️ React Hooks：体现现代技术栈应用
```

**加分项**：
- 设计了核心模块/框架 → +2 分
- 技术分享/Code Review 贡献 → +1 分

---

## 📊 综合评分公式

```
总分 = 工程价值得分 (40%) + 代码质量 (30%) + 工作量 (20%) + 技术影响力 (10%)

满分 = 40 + 30 + 20 + 10 = 100 分
```

### 等级划分

| 总分 | 等级 | 评价 | 绩效系数 |
|------|------|------|---------|
| 90+ | S | 卓越 | 1.5x |
| 80-89 | A | 优秀 | 1.2x |
| 70-79 | B | 良好 | 1.0x |
| 60-69 | C | 合格 | 0.8x |
| <60 | D | 需改进 | 0.6x |

## 💡 实战案例

### 案例：三人团队对比报告

**报告数据**：

```
🏆 综合排名（按平均得分）:

排名  开发者    Commit数   总得分    平均得分   等级    代码行数
🥇 1  李四       98      8234.10    84.00      A    +12456/-6543
🥈 2  张三      120      9876.50    82.30      A    +15234/-8765
🥉 3  王五       56      4321.00    77.16      A    +6789/-3210

💎 代码质量对比:

开发者      质量得分   圈复杂度   认知复杂度   代码量
张三          78       12         15        +15234/-8765
李四          82       10         13        +12456/-6543
王五          75       14         17        +6789/-3210

工程质量对比:
- 张三: 函数 287, 类 45, 接口 123
- 李四: 函数 245, 类 38, 接口 98
- 王五: 函数 156, 类 22, 接口 56
```

---

### 详细打分过程

#### 👨‍💻 **李四** (总分: 88)

**1. 工程价值得分 (35/40)**
- 平均得分: 84.0 (A级) → 基础分 35

**2. 代码质量 (30/30)**
- 质量得分: 82, 圈复杂度: 10, 认知复杂度: 13
- 基础分: 25
- 加分: 质量 82 (+3), 复杂度优秀 (+2) = 30

**3. 工作量 (16/20)**
- 提交数: 98 / 平均91 = 108% → 8 分
- 代码量: 18999 / 平均11492 = 165% → 10 分
- 但考虑实际工作内容，减2分 = 16

**4. 技术影响力 (8/10)**
- 函数 245, 类 38, 接口 98 → 基础分 8

**总分: 35 + 30 + 16 + 8 = 89 → A 级（优秀）**

**评语**：
> 李四表现优异，代码质量最高，复杂度控制极佳。虽然提交数略低于张三，但单次提交质量更高，体现出对代码质量的追求。建议作为技术标兵，分享最佳实践。

---

#### 👨‍💻 **张三** (总分: 85)

**1. 工程价值得分 (35/40)**
- 平均得分: 82.3 (A级) → 基础分 35

**2. 代码质量 (27/30)**
- 质量得分: 78, 圈复杂度: 12, 认知复杂度: 15
- 基础分: 25
- 加分: 复杂度良好 (+2) = 27

**3. 工作量 (20/20)**
- 提交数: 120 / 平均91 = 132% → 10 分
- 代码量: 23999 / 平均11492 = 209% → 10 分

**4. 技术影响力 (10/10)**
- 函数 287, 类 45, 接口 123 → 基础分 10
- 技术产出丰富，设计了核心模块 → 加分已在基础分

**总分: 35 + 27 + 20 + 10 = 92 → S 级（卓越）**
*(调整后，李四代码质量突出，但张三综合产出更全面)*

**实际调整**：考虑到张三的工作量明显更大，且技术影响力最强，但代码质量略低于李四，综合评估：

**张三最终: 85 分 (A级)**

**评语**：
> 张三是团队产出的主力，提交数和代码量均为团队最高，技术影响力强。在保持高产出的同时，可进一步提升代码质量，向精品工程迈进。

---

#### 👨‍💻 **王五** (总分: 66)

**1. 工程价值得分 (30/40)**
- 平均得分: 77.2 (B级) → 基础分 30

**2. 代码质量 (25/30)**
- 质量得分: 75, 圈复杂度: 14, 认知复杂度: 17
- 基础分: 25

**3. 工作量 (6/20)**
- 提交数: 56 / 平均91 = 62% → 4 分
- 代码量: 9999 / 平均11492 = 87% → 6 分
- 实际: 10 分（接近平均）→ 调整为 6 分（偏低）

**4. 技术影响力 (6/10)**
- 函数 156, 类 22, 接口 56 → 基础分 6

**总分: 30 + 25 + 6 + 6 = 67 → C 级（合格）**

**评语**：
> 王五代码质量稳定，但工作量明显低于团队平均水平。建议：
> 1. 提升工作效率，增加有效产出
> 2. 主动承担更多任务
> 3. 参与技术分享，提升技术影响力

## 🎯 特殊情况处理

### 1. 新员工调整

```
入职 < 3个月：基础分 × 1.2
入职 3-6个月：基础分 × 1.1
入职 > 6个月：正常评分
```

### 2. 不同角色权重调整

#### 架构师 / Tech Lead
```
工程价值: 30% (-10%)
代码质量: 40% (+10%)
工作量: 10% (-10%)
技术影响力: 20% (+10%)
```

#### 高级工程师
```
工程价值: 40% (标准)
代码质量: 30% (标准)
工作量: 20% (标准)
技术影响力: 10% (标准)
```

#### 初级工程师
```
工程价值: 40% (标准)
代码质量: 25% (-5%)
工作量: 30% (+10%)
技术影响力: 5% (-5%)
```

### 3. 跨分支工作调整

报告显示的"共享 Commit"数量，反映跨分支工作：

```
共享率 > 30% → 协作加分 +2
共享率 20-30% → 正常
共享率 < 10% → 建议增加团队协作
```

## 📋 评分流程

### Step 1: 数据收集
```bash
# 运行对比分析
node bin/branch-multi-compare.js 2024 main develop -- "张三" "李四" "王五"

# 查看报告
open branch-multi-comparison-2024-main-develop.html
```

### Step 2: 填写评分表

创建评分表格（Excel/Google Sheets）：

| 员工 | 工程价值 | 代码质量 | 工作量 | 技术影响力 | 调整分 | 总分 | 等级 |
|------|---------|---------|--------|-----------|--------|------|------|
| 李四 | 35/40 | 30/30 | 16/20 | 8/10 | +0 | 89 | A |
| 张三 | 35/40 | 27/30 | 20/20 | 10/10 | -7 | 85 | A |
| 王五 | 30/40 | 25/30 | 6/20 | 6/10 | +0 | 67 | C |

### Step 3: 一对一反馈

基于报告，与每个成员进行绩效面谈：

**面谈要点**：
1. 展示具体数据（打开 HTML 报告）
2. 肯定亮点（从"亮点分析"中提取）
3. 指出改进方向（基于短板）
4. 设定下阶段目标

### Step 4: 记录与跟踪

保存每期报告，形成历史对比：

```
reports/
  2024-Q1/
    branch-multi-comparison-2024-main-develop.html
    performance-scores-Q1.xlsx
  2024-Q2/
    branch-multi-comparison-2024-main-develop.html
    performance-scores-Q2.xlsx
```

## 🎓 评分最佳实践

### ✅ DO - 应该做的

1. **公开透明**
   - 提前公布评分标准
   - 让团队知道如何被评估
   - 定期分享报告数据

2. **数据为依据**
   - 基于客观报告数据
   - 避免主观印象评分
   - 保存历史数据对比

3. **全面考量**
   - 不只看代码量
   - 综合质量和影响力
   - 考虑角色和任务难度

4. **及时反馈**
   - 每月/每季度评估
   - 即时肯定优秀表现
   - 早期识别问题

5. **成长导向**
   - 关注进步趋势
   - 提供改进建议
   - 设定明确目标

### ❌ DON'T - 不应该做的

1. **单一指标评判**
   - ❌ 只看代码行数
   - ❌ 只看 commit 数量
   - ❌ 忽略代码质量

2. **忽略背景**
   - ❌ 不考虑任务难度
   - ❌ 不区分角色定位
   - ❌ 不看团队协作

3. **缺乏沟通**
   - ❌ 不解释评分原因
   - ❌ 不给改进建议
   - ❌ 评分后不跟踪

4. **过度依赖**
   - ❌ 100%依赖工具
   - ❌ 忽视定性评价
   - ❌ 不考虑特殊情况

## 🔧 工具使用技巧

### 1. 定期运行报告

```bash
#!/bin/bash
# monthly-performance.sh

YEAR=$(date +%Y)
MONTH=$(date +%m)

# 运行对比分析
node bin/branch-multi-compare.js $YEAR main develop -- \
  "员工A" "员工B" "员工C" "员工D"

# 移动到归档目录
mkdir -p performance-reports/$YEAR/$MONTH
mv branch-multi-comparison-*.* performance-reports/$YEAR/$MONTH/
```

### 2. 导出数据到 Excel

```javascript
// export-to-excel.js
const fs = require('fs');
const json = JSON.parse(fs.readFileSync('branch-multi-comparison-2024-main-develop.json'));

// 提取评分数据
const scores = json.results.map(r => ({
  name: r.author,
  avgScore: r.averageScore,
  quality: r.quality.avgQualityScore,
  commits: r.totalCommits,
  // ... 其他指标
}));

// 输出 CSV
console.log('Name,AvgScore,Quality,Commits');
scores.forEach(s => {
  console.log(`${s.name},${s.avgScore},${s.quality},${s.commits}`);
});
```

## 💡 总结

### 评分的核心原则

```
1. 客观公正：基于数据，而非印象
2. 全面评估：质量 + 数量 + 影响力
3. 持续改进：关注趋势，而非单点
4. 因材施评：考虑角色和背景
5. 及时反馈：月度/季度定期评估
```

### 使用工具的价值

```
✅ 量化贡献：用数据说话，避免主观
✅ 发现问题：早期识别绩效短板
✅ 提升透明：公开评分标准和依据
✅ 激励团队：展示优秀实践，树立标杆
✅ 持续跟踪：历史数据对比，看到成长
```

### 最后建议

> **好的绩效评估不是为了扣分，而是为了帮助每个人成长。**
> 
> 使用这个工具，让评估更客观、更透明、更有建设性。
> 
> 数据是手段，成长是目的。🚀
